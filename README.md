# Задание 1: Разделение фронтенда Mesto на микрофронтенды

В каталоге components расположены js-компоненты клиентского приложения. 
Каталог blocks содержит стили. Каталог utils содержит служебные скрипты.

Компоненты:
1. Компонент приложения App ()
1. Компонент регистрации Register (роут /sigup)
1. Компонент логина Login (роут /signin)
1. Поп-ап результата регистрации InfoTooltip
1. Хэдер Header (получает от App функцию-колбек на разлогин)
1. Страница Main (Пользователь, кнопка и список карточек, отображается только для аутентифицированного пользователя)
1. Компонент карточки Card (набор карточек формируется по данным пользователя)
1. Футер Footer
1. Попап подтверждения PopupWithForm (используется в AddPlacePopup, EditAvatarPopup, EditProfilePopup) 
1. Поп-ап редактирования профиля EditProfilePopup
1. Поп-ап редактирования аватара EditAvatarPopup
1. Поп-ап добавления места AddPlacePopup
1. Поп-ап отображения изображения ImagePopup

Приложение построено на основе React. Также использует библиотеку для снятия веб-метрик web-vitals

В приложении явно выделено два файла с утилитами: api.js и auth.js. Утилита api.js абстрагирует
взаимодействие с бизнес-API бэкенда. auth.js абстрагирует взаимодействие с API аутентификации
и регистрации. Также есть файл CurrentUserContext.js для получения текущего контекста приложения в
компонентах App, Card, EditProfilePopup и Main (получение пользователя). Также есть скрипт index.js,
который выступает точкой начала построения приложения. Также есть скрипт serviceWorker.js, который
добавляет возможность ускорения и оффлайн-работы, но не используется. Также есть утилитарный компонент 
ProtectedRoute, служащий для перенаправления незалогиненых пользователей на страницу логина.

В приложении используются следующий файлы стилей:
1. Каталог auth-form. Используются в Register и Login
1. Каталог card. Используются в Card
1. Каталог content. Используются в Main
1. Каталог footer. Используются в Footer
1. Каталог header. Используются в Header
1. Каталог login. Не используются.
1. Каталог page. Используются в компонентах Footer, Header и Main
1. Каталог places. Используются в Card и Main
1. Каталог popup. Используются в AddPlacePopup, EditAvatarPopup, EditProfilePopup, ImagePopup, InfoTooltip, PopupWithForm
1. Каталог profile. Используются в Main

На основании сделанных наблюдение можно выделить следующие составляющие:
1. Авторизация и регистрация. Это служебная функциональность, которая и на 
данный момент выделена в коде отдельными скриптами утилит. Включает в себя компоненты 
регистрации и логина, а также поп-ап результатов регистрации InfoTooltip,
утилиту auth.js, стили auth-form. Также включает часть компонента App, связанную с 
состоянием логина. Целесообразно выделить эту составляющую в отдельный
микрофронт, т.к. соответствующая часть бэкенда может быть выделена в компонент
Identity and Access Management. Эта часть приложения слабо связана с прочими частями,
предоставляя им токен для определния информации о пользователе.
1. Профиль пользователя. Сейчас это часть компонента Main, которую потребуется выделить в
самостоятельный компонент. Включает в себя поп-апы EditAvatarPopup и EditProfilePopup,
стили profile. Также включает часть api.js, относящуюся к работе с пользователем.
Профиль пользователя целесообразно отделить в самостоятельную часть приложения,
чтобы сконцентрировать в ней вопросы работы с пользовательскими данными.
1. Каталог карточек. Включает в себя компонент Main за вычетом профиля пользователя, компонент
Card, поп-апы AddPlacePopup, ImagePopup, стили card, content, places. Также включает 
часть api.js, относящуюся к работе с карточками. Это часть приложения, которая несёт
основную значимость для бизнеса. В части каталога карточек можно сделать замечание, что также 
есть возможность выделить составляющую "лайки на фотографиях", однако это должно быть обосновано, 
например, аналогичным разделением на стороне бэкенда, различными требованиями к критичности отказа
функциональности и т.п. Поскольку на данный момент таких оснований не просматривается,
то я буду далее рассматривать каталог карточек как одну составляющую приложения, а
разделение её на функциональность работы с карточками и на функциональность лайков
буду рассматривать как вариант развития сервиса. 
1. Оболочка. Отвечает за сборку страницы и за общие стили. Включает компоненты App, Header, 
Footer, PopupWithForm? CurrentUserContext. Включает стили footer, header, page, popup. 

Поскольку сервис Mesto - простой сервис, то в качестве метода интеграции целесообразно
использовать интеграцию build-time, чтобы обеспечить простоту развёртывания и не 
создавать для пользователя дополнительные задержки UI на интеграцию в runtime.

В качестве технического решения для реализации микрофронтов предлагаю использовать
библиотеку Webpack Module Federation, чтобы обеспечить доступ микрофронтов к
общим компонентам (общие поп-апы). Основную особенность альтернативной технологии Single SPA - 
независимость от фреймворка - на данный момент не удастся использовать эффективно, т.к. приложение
слишком простое и усложнение его реализации за счёт внедрения разных фреймворков
повлечет усложнение дальнейшей разработки. 

Взаимодействие между микрофронтендами предлагаю реализовать на основе библиотеки
глобального состояния. На данный момент такое состояние - это токен пользователя,
который будет отдавать микрофронтенд авторизации, а пользоваться токеном будут
- Оболочка - для отображения логина пользователя в хэдере
- Профиль пользователя - для отображения данных о пользователе
- Каталог карточек - для отображения карточек пользователя

Примерная схема изменений отражена в каталоге (microfrontend)[./frontend/microfrontend/]

В задании выполнены пункты "Проектирование" и "Планирование изменений". Пункт "Запуск готового кода"
не выполнен.

# Задание 2: Декомпозиция веб-приложения на Django на микросервисы

Диаграмма с решением: [task2.drawio](./task2/task2.drawio).

## Пояснение решения

Из исходной диаграммы можно выделить следующие сущности, относящиеся к бизнес-домену:
1. Профиль пользователя
1. Аукцион
1. Ставка аукциона
1. Товар
1. Услуга
1. Заказ
1. Платежная операция
1. Отчет по продажам
1. Отчет активности пользователей
1. Заявка в техподдержку (включая заявки на апелляцию)

Между выявлены сущностями следующие зависимости
1. *Профиль пользователя* содержит список *аукционов*
1. *Профиль пользователя* содержит список *ставок на аукционе*
1. *Профиль пользователя* содержит список *товаров*
1. *Профиль пользователя* содержит список *услуг*
1. *Профиль пользователя* содержит список *заказов*
1. *Профиль пользователя* соответствует списку *платежных операций*
1. *Профиль пользователя* содержит список *отчетов по продажам*
1. *Профиль пользователя* содержит список *заявок в техподдержку*
1. *Товар* соответствует списку *аукционов*
1. *Аукцион* содержит список *ставок на аукционе*
1. *Аукцион* соответствует *платежной операции*
1. *Услуга* соответствует списку *заказов*
1. *Заказ* соответствует *платежной операции*

Действуют следующие акторы:
1. Клиент
1. Администратор
1. Специалист поддержки
1. Платежный сервис

Система позволяет следующие варианты использования:
1. *Клиент* создаёт/редактирует *профиль пользователя*
1. *Администратор* валидирует созданный *профиль пользователя*
1. *Клиент* создаёт/редактирует/сохраняет/подтверждает/размещает/просматривает список/
просматривает детали/получает нотификацию/удаляет/отображает статистику *заказов*
1. *Клиент* добавляет/изменяет/удаляет/ищет/фильтрует/сортирует *услугу*
1. *Клиент* добавляет/изменяет/удаляет/ищет/фильтрует/сортирует *товар*
1. *Клиент* создаёт/заявляется/обновляет ставку/обновляет статус/получает нотификацию об *аукционе*
1. *Администратор* редактирует *аукцион*
1. *Клиент* генерирует *отчет по продажам*
1. *Клиент* инициирует/просматривает список/просматривает статус/запрашивает статус у платежной системы/отменяет *платежную операцию*
1. *Клиент* создаёт *заявку в техподдержку*
1. *Специалист поддержки* обновляет статус *заявки в техподдержку*
1. *Администратор* генерирует *отчет активности пользователей*

Исходя из сказанного, предлагаю следующее разделение на функциональные домены:
1. Профиль пользователя
1. Каталог товаров и услуг
1. Ведение заказов
1. Ведение аукционов
1. Ведение платежей
1. Техподдержка
1. Сервис авторизации

Для каждого домена на диаграмме выделен соответствующий модуль. Также
на диаграмме более детально отражены взаимодействия между модулями, а также взаимодействия
акторов и системы.

Дополнительно на диаграмму добавлен новый модуль шлюза API Gateway, который служит единой точкой входа для
клиентских запросов. Также модуль шлюза тесно взаимодействует с Сервисом авторизации для обеспечения
большего уровня безопасности. 

Предлагаю сохранить технологический стек Django/Python для выделенных микросервисов, следуя
логике того, что в ИТ отделе уже существует компетенция для данного стека и его целесообразно оставить.

Для Сервиса авторизации предлагаю использовать готовое решение Keycloack для учета пользователей.
Это брокер авторизаций, широко используемый в индустрии. Для эффективного взаимодействия с ним
потребуется написать простой сервис-адаптер.

Пользовательские данные предлагаю сосредоточить в сервисе Профиль пользователя. Также ответственностью
этого сервиса будет взаимодействие с Клиентом в части нотификаций.

Для учета товаров и услуг, предоставляемых Клиентами на платформе предлагаю использовать модуль
Каталог товаров и услуг. Модуль должен обеспечивать ведение Клиентом собственных товаров и услуг, а также
поиск по товарам и услугам других Клиентов.

Обеспечение проведения аукционов предлагаю реализовать в модуле Ведение аукционов. Доступ к модулю со стороны
клиентов Обеспечивается через Портал электронной площадки. Доступ к модулю со стороны Администраторов Сервиса
осуществляется из сети LAN. В качестве дальнейшего развития следует рассмотреть возможность доступа через
специализированное автоматизированное рабочее место из WAN. 

Выполнение заказов предлагаю реализовать в модуле Ведение заказов.

Модуль Ведение платежей должен выполнять функции проведения платежей по заказам и проведенным аукционам.
Также модуль взаимодействует с Платежными системами. В качестве дальнейшего развития целесообразно
рассмотреть возможность выделения адаптеров под различные Платежные системы в отдельные модули.
Это позволит упростить подключение дополнительных Платежных систем за счет изоляции логики работы с
конкретной Платежной системой, позволит выполнять раздельное масштабирование ресурсов.

Модуль Отчетность обеспечивает формирование отчетов. На диаграмме модуль взаимодействует с модулем
Ведение платежей для построения отчета по продажам, а с модулем Сервис авторизации для построения отчета пользовательской активности. Это сделано из предположения, что такие сведения требуются для построения отчетов. 
При необходимости диаграмму следует доработать связями и с другими модулями, если в отчет
потребуется включить дополнительную информацию.

Модуль Техподдержка обеспечивает обработку заявок в техподдержку. Доступ, как и в случае с модулем
Ведение аукционов, для Клиента обеспечивается посредством Портала электронной площадки, а для
специалиста Техподдержки - напрямую из локальной сети LAN. В качестве направления развития
также для специалиста техподдержки в будущем возможна реализация специализированного
автоматизированного рабочего места (АРМ).

На диаграмме отмечено наличие отдельных БД для модулей. При переходе от общей БД к набору
индивидуальных БД для каждого модуля следует учесть механизмы обеспечения консистентности
данных. Можно выделить несколько сценариев, при которых консистентность будет нарушена:
1. Клиент передумал выполнять оплату по назначенному заказу или выигранному аукциону.
2. Произошел технический сбой в модуле Проведение платежей или на стороне Платежной системы, 
приведший к невозможности провести оплату.
3. Администратор отредактировал аукцион и отменил результаты аукциона.

В каждом из этих случаев следует отразить факт выполнения альтернативного сценария не только
в модуле Ведение платежей, но и в модулях Ведение заказов или Ведение аукционов. Если следовать
паттерну Sage, то требуется выполнить компенсирующие операции. Поскольку таких операций немного
(отмена заказа и отмена результатов аукциона), то предлагаю использовать прямые вызовы от одного
модуля к другому. С ростом количества таких операций можно рассмотреть переход к событийной модели
оповещения, например.
